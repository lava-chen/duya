# Structured Brainstorming

A systematic approach to creative problem-solving that balances idea generation with practical evaluation.

---

## Core Principle

**Brainstorming is not about having good ideas — it's about having many ideas, then finding the good ones.**

Quality emerges from quantity + structured evaluation.

---

## Phase 1: Problem Framing

### Define the Challenge

Before generating ideas, be crystal clear on:

```
PROBLEM STATEMENT TEMPLATE

We need to [action] for [target audience] 
so that they can [benefit/outcome],
but [constraint/limitation].

Example:
"We need to reduce onboarding time for new users 
so that they can experience value within 5 minutes,
but we can't add more support staff."
```

### Reframe as "How Might We"

Convert problems into opportunity statements:

| Problem Frame | HMW Frame |
|---------------|-----------|
| "Users don't understand our pricing" | "How might we make pricing transparent at a glance?" |
| "Our app is too slow" | "How might we make every interaction feel instant?" |
| "Team communication is broken" | "How might we make remote collaboration feel like being in the same room?" |

**Why HMW works:**
- Assumes a solution exists (optimistic)
- Opens multiple solution paths
- Invites collaboration
- Removes blame

## Phase 2: Divergent Thinking (Generate)

### Rule of Divergence

During idea generation:
- **Defer judgment** — No criticism, no "but"
- **Quantity over quality** — Aim for 20+ ideas
- **Build on others** — "Yes, and..." not "Yes, but..."
- **Encourage wild ideas** — They spark practical ones
- **One conversation at a time** — Everyone heard

### Framework 1: SCAMPER

Systematically modify an existing solution:

| Letter | Action | Question |
|--------|--------|----------|
| S | Substitute | What can we replace? (materials, people, processes) |
| C | Combine | What can we merge? (features, services, teams) |
| A | Adapt | What can we borrow from other contexts? |
| M | Modify | What can we magnify, minimize, or change? |
| P | Put to other uses | Where else could this work? |
| E | Eliminate | What can we remove? |
| R | Reverse/Rearrange | What if we did the opposite? |

**Example (improving a coffee shop):**
- Substitute: Replace cashiers with app ordering
- Combine: Coffee + co-working space
- Adapt: Library quiet hours for focused work
- Modify: 24-hour service for night owls
- Put to other uses: Event space after hours
- Eliminate: No seating, purely grab-and-go
- Reverse: Customers make their own coffee

### Framework 2: Assumption Reversal

List all assumptions, then reverse them:

```
ASSUMPTIONS ABOUT: [Problem]

1. Users need [X]
   → Reverse: Users don't need [X], they need [Y]
   → Idea: [New approach]

2. The solution must be [Y]
   → Reverse: The solution must NOT be [Y]
   → Idea: [Opposite approach]

3. This requires [Z]
   → Reverse: This requires no [Z]
   → Idea: [Zero-resource approach]
```

### Framework 3: Analogical Thinking

Borrow from unrelated domains:

```
SOURCE DOMAIN → TARGET DOMAIN

Nature → Architecture (biomimicry)
Gaming → Education (gamification)
Music → Product Design (rhythm, flow)
Cooking → Software (recipes as code)
Sports → Team Management (playbooks)
```

**Process:**
1. Pick a random domain
2. Identify its key mechanisms/principles
3. Map to your problem
4. Generate ideas from the mapping

### Framework 4: Constraint Removal

Imagine constraints don't exist, then add back:

```
STEP 1: Unlimited resources
"If we had infinite budget, time, and talent..."

STEP 2: Identify the core value
"What specifically makes this idea great?"

STEP 3: Constrain creatively  
"How can we achieve that core value with our actual limits?"
```

### Framework 5: Worst Idea First

Generate deliberately bad ideas, then invert:

```
BAD IDEAS FOR: [Problem]

1. Make it as complicated as possible
   → Invert: What's the simplest possible version?

2. Make it extremely expensive
   → Invert: How could this be free?

3. Make users wait as long as possible
   → Invert: How can this be instant?

4. Require maximum effort from users
   → Invert: How can this require zero effort?
```

## Phase 3: Convergent Thinking (Select)

### Clustering

Group ideas by theme:

```
IDEA CLUSTERS

Cluster A: Technology Solutions
- Idea 1
- Idea 4
- Idea 12

Cluster B: Process Changes
- Idea 2
- Idea 7
- Idea 15

Cluster C: Partnership Approaches
- Idea 3
- Idea 9
```

### Evaluation Matrix

Score ideas across key criteria:

```
Criteria:    Impact  Effort  Risk  Cost  Speed  TOTAL
Idea A:      5       2       4     3     5      19
Idea B:      4       4       3     4     4      19
Idea C:      3       5       5     5     3      21

Scale: 1-5 (5 = best for Impact/Speed/Cost, 5 = lowest for Effort/Risk)
```

### The 2x2 Matrix

Map ideas on two key dimensions:

```
                High Impact
                     |
    Quick Wins  ●    |    ●   Major Projects
    (Do first)       |         (Plan carefully)
                     |
    -----------------+------------------
    Fill-ins      ●  |  ●   Thankless Tasks
    (Do if time)     |       (Avoid)
                     |
                Low Impact
                     
    ← Low Effort              High Effort →
```

### Selection Criteria

**Must-have criteria** (eliminate if missing):
- Feasible with current resources
- Addresses the core problem
- Aligns with constraints

**Nice-to-have criteria** (rank by these):
- Speed to implement
- Potential impact
- Uniqueness/differentiation
- Learning value
- Reversibility (can we undo it?)

## Phase 4: Idea Development

### Concept Refinement

Take top ideas and develop:

```
CONCEPT: [Idea name]

Core concept (1 sentence):
[What it is]

How it works (3-5 steps):
1. [Step 1]
2. [Step 2]
3. [Step 3]

Key assumptions:
- [Assumption 1]
- [Assumption 2]

What would need to be true:
- [Condition 1]
- [Condition 2]

Quick test:
[How to validate in 1 week]
```

### Pre-Mortem

Before committing, imagine it failed:

```
PRE-MORTEM: [Idea name]

"It's 6 months later and [idea] completely failed. Why?"

Possible reasons:
1. [Reason 1] → Mitigation: [Action]
2. [Reason 2] → Mitigation: [Action]
3. [Reason 3] → Mitigation: [Action]
```

## Common Pitfalls

### During Divergence
- **Premature convergence**: "That won't work" too early
- **Groupthink**: Following the loudest voice
- **Anchoring**: Getting stuck on first idea
- **Expert bias**: "We've tried that before"

### During Convergence
- **Analysis paralysis**: Over-scoring, under-deciding
- **Sunk cost**: Keeping ideas because of time invested
- **False consensus": Assuming silence = agreement
- **Perfectionism": Waiting for the perfect idea

### General
- **Wrong problem**: Solving symptoms, not causes
- **Solution fixation": Falling in love with one approach
- **Scope creep": Expanding the problem mid-session

## Session Formats

### Solo Brainstorming (30 min)
1. Problem framing (5 min)
2. Rapid ideation using SCAMPER (15 min)
3. Cluster and select (10 min)

### Pair Brainstorming (45 min)
1. Problem framing together (10 min)
2. Individual ideation (10 min)
3. Share and build (15 min)
4. Evaluate and select (10 min)

### Group Brainstorming (60 min)
1. Warm-up exercise (5 min)
2. Problem framing (10 min)
3. Round-robin ideation (20 min)
4. Clustering (10 min)
5. Evaluation (10 min)
6. Next steps (5 min)

## When to Use Which Framework

| Situation | Recommended Framework |
|-----------|----------------------|
| Improving existing solution | SCAMPER |
| Breaking out of conventional thinking | Assumption Reversal |
| Need fresh perspective | Analogical Thinking |
| Feeling resource-constrained | Constraint Removal |
| Group is stuck | Worst Idea First |
| Complex problem with many facets | All frameworks in sequence |
