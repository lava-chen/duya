**v0.2.0-beta.1** (2026-06-26) — 92 commits since `v0.1.5`. First beta on the
`0.2.x` line. Headline: the **Codex-aligned chat composer** (inline
markdown images, Esc-interrupt with a "Stopped" badge, and the
slash-command popover now inserts skills directly into the message
text instead of rendering a detached badge), the **Agent Mailbox**
(Plan 202) data layer + UI, the **`PermissionRequiredError` typed
permission flow**, the **multi-provider migration** (Plans 203 / 204
/ 205 / 209), the new **right-side panel v4** with a multi-tab page
registry, **node-pty terminal** as a PanelZone page, **DOCX/PPTX/XLSX
preview workspaces**, the **`.ipynb` Read integration**, the
**AgentServer `failSession` hardening**, and the first batch of
`@duya/cli` plugin control-plane write ops (`enable` / `disable` /
`doctor`).

This is a **beta**: the Mailbox UI and the v4 panel are new and
still being smoke-tested. We tag now so early adopters can shake
out the packaging + first-run experience before the next stable
release.


**Highlights**


- **Codex-aligned chat composer (Plan 208 + slash-command popover
  rewrite)** — `MessageItem` now renders inline markdown images
  via the new `MarkdownImage` component (click-to-zoom into the
  existing lightbox), bare image URLs are preprocessed into
  `![](url)` so the assistant can drop a plain link and still get
  a thumbnail, and the slash-command popover is split into
  Settings / Skills sections with `CubeIcon` fallback. Skills
  resolve through the new `insert_slash_command` action: the
  active slash fragment is replaced in the message text with the
  full command + trailing space, then the cursor is restored after
  the command end. The detached `CommandBadge` path for skills is
  gone.
- **Esc-interrupt + Stopped badge (P2-β)** — pressing **Esc** while
  streaming now stops the agent and flags the partial assistant
  message as `metadata.interrupted` (local-only, never persisted).
  `MessageItem` shows a small "Stopped" badge with an i18n tooltip;
  the global Esc handler is gated on `isStreaming` and skips
  editable targets so it doesn't fight the slash-menu popover.
- **Agent Mailbox (Plan 202)** — Codex-like in-run instruction
  pipeline. PR1 ships the data layer (`mailbox_*` tables, claim /
  lease / checkpoint, soft interrupt) and the agent-server
  hardening. The chat UI gets a `MailboxPanel`, `MailboxBubble`
  rows, an inline `MailboxComposer`, and a `StreamingInputBar`
  shown in place of `MessageInput` during streaming. PR2 wires the
  agent-side `MailboxService` + `before_model_turn` absorption.
- **`PermissionRequiredError` typed permission flow** — the legacy
  `<tool_use_permission_required>` string sentinel is replaced by a
  typed error class. `BaseTool.execute()` can throw to signal that
  it needs explicit user permission; the executor routes through
  `ToolUseContext.requestPermission`. `PermissionCheckResult`
  gains a `mode` discriminator (`'generic' | 'ask_user_question' |
  'exit_plan_mode'`) so tools with their own dedicated UI declare
  which panel the executor should mount — `AskUserQuestionTool`
  now goes through the dedicated `AskUserQuestionUI` panel rather
  than the generic Allow/Deny prompt.
- **Multi-provider migration (Plans 203 / 204 / 205 / 209)** —
  `AppConfig` adds a soft `defaultProviderId` and a
  `multi-provider-v1` migration. The renderer drops the `isActive`
  filter in 6 places and gates visibility on `isDefault + hasApiKey`.
  The settings provider UI is split into a 4-layer architecture
  (Query → Hook-per-concern → Orchestrator → Wiring):
  `ProvidersSection` shrinks from a 1066 LoC monolith to ~80 LoC,
  with the per-card list / add / edit / delete / test / quota
  actions moved to a `ProviderManagement` orchestrator. Inline edit
  page replaces the `ProviderConnectDialog` modal with a two-step
  flow: a preset card grid (`provider-picker`) → a single-provider
  form (`provider-edit`). The Plan 209 masked-key bug (saving an
  edit without touching the key field persisted the masked
  placeholder, silently destroying the real credential) is fixed
  with a 3-state `useApiKeyState` machine and server-side defense
  in depth in `provider-store.upsertLlmProvider`.
- **Right-side panel v4** — the side panel is rewritten as a
  multi-tab page registry. `usePanel` exposes a `tabs / openPage /
  openOrActivatePage / activateTab / closeTab` API; `PanelHeader`,
  `PanelZone`, `PagePickerMenu`, and the `FileTreePanel` /
  `ResearchModePanel` / `SidebarConductorView` consumers are wired
  through the new model. `TaskListPanel` is replaced by the new
  `TaskDrawer` and the sub-agent row is de-duplicated.
- **node-pty terminal** — `Terminal` is integrated as a PanelZone
  page with `xterm.js`. The chat input supports terminal reference
  chips via `RichTextInput`.
- **Office + File Preview workspaces (Plans 215 / 216)** — the
  right sidebar gains compact DOCX / PPTX / XLSX preview panes
  and an expanded read-only file preview workspace with tabs, a
  project tree, and a persistent composer. `FilePreviewPanel` and
  `OfficePanel` share the chat layout, with selection context fed
  into the Agent. `files:preview` IPC + integration specs ship in
  `e2e/ipc/files.spec.ts`.
- **`.ipynb` Read integration** — the `Read` tool routes Jupyter
  notebooks through a new `NotebookParser` (sidecar image
  extraction, `cell_range` parameter, `.ipynb` snapshot test).
- **BrowserTool zod/v4 schema fix** — `z.preprocess()` compiles to
  a `ZodPipe` on zod/v4; the previous generator descended via
  `def.schema` and returned the string `'pipe'`, falling through
  to `{ type: 'string' }` and silently telling the LLM that `urls`
  was a string. The generator now walks `def.out` for preprocessed
  fields. Defensive `z.preprocess` wrappers are added to `urls`,
  `useBrowser`, `timeoutMs`, `frameIndex`, `fullPage`,
  `interactiveOnly`, and `maxLength`. Regression tests pin the
  fix.
- **`@duya/cli` plugin control plane (Plan 100 complete)** —
  `duya plugin list [--enabled] [--verbose] [--format tsv|json]`,
  `info`, `enable --yes`, `disable --yes`, `doctor` (5 checks; exit
  1 on `error`). Audit events are stamped to
  `control-plane-audit.log.jsonl` with `X-Correlation-Id`.
- **AgentServer `failSession` hardening** — `SessionManager` gains
  a one-call `failSession(id, message, retryable?)` that records
  retryable metadata and transitions the session to `ERROR` from
  both `STREAMING` and `COMPLETING`. The router calls it on every
  failure path (worker `error`, worker `exit` before close, SSE
  `error` frame, GET `/chat` replay). `VALID_TRANSITIONS` allows
  `COMPLETING → ERROR`. A worker crash mid-completion is no longer
  stuck on `COMPLETING`.
- **`@duya/conductor` package** — the canvas renderer moves out of
  `src/` into `@duya/conductor/renderer`; `Plan 70-74` and the
  MindMap / Frame / Toolbar interactions are wired through the new
  package boundary; the iterative visual refinement pass lands.

**Internals**

- `useApiKeyState` is a 3-state machine: `untouched` →
  `replaced` → `cleared`. The 2-state version conflated the
  server-provided mask with user-typed input and was the root
  cause of Plan 209.
- `useApiKeyLink` shares the mask hint with the chat-side key
  entry path.
- `useProviderEditSave` carries an `apiKey?: string` contract:
  `undefined` (keep current) vs `''` (clear) vs a new value.
- `src/lib/providers/secret.ts` is a shared `isMaskedKey` /
  `isLikelyRealApiKey` module imported by both the renderer and
  the main process.
- `src/lib/providers/canCheckQuota.ts` is a shared predicate for
  "this provider exposes a quota API".
- `src/lib/subagent-result.ts` is a shared
  `parseSubAgentToolResult` parser.
- `src/lib/chat-file-links.ts` is a shared `openLocalFileTarget` /
  `isLikelyLocalFileReference` helper.
- `useAdaptiveTypewriter` is a shared hook (was inlined in
  `StreamingMessage`).
- `ModelInputWithFetch` (settings/forms/shared) is the reusable
  model input used by both the chat-side picker and the new
  `ProviderEditView`.
- `Message.metadata` bag is added (renderer-only, never persisted)
  for local-only flags like `interrupted`.
- `PopoverItem.group` (`'settings' | 'skills'`) and the
  `insert_slash_command` `InsertResult` variant replace the legacy
  `set_badge` / `immediate_command` branches.

**Upgrade notes**

- `provider:active` / `context.activeProviderId` is deprecated;
  rename to `provider:default` / `context.defaultProviderId`.
  The `multi-provider-v1` migration handles the rename on startup
  — no manual action is required.
- The single-active-provider model (`isActive` filter) is gone.
  Visibility gates on `isDefault + hasApiKey`; the new per-row
  "Set as default" action replaces the legacy "Enable" /
  "In use" toggle.
- Gateway sessions now hardcode
  `permission_profile = 'default'`. If you previously relied on
  `gatewayPermissionMode` / desktop `permissionMode` to gate
  IM-channel sessions, the IM channel is now controlled by the
  platform-specific pairing / allowlist flow instead.
- The `ProviderConnectDialog` modal is gone from the settings
  flow. The settings page now mounts `ProviderPickerView` and
  `ProviderEditView` as inline pages; the dialog is still used
  by the onboarding flow.
- Plan 209's `useApiKeyState` 3-state machine means saving an
  edit without touching the key field now keeps the existing
  credential (was previously the masked-key bug that silently
  overwrote the real key).
- The detached `CommandBadge` path for slash skills is gone.
  Skill selections now insert the full command directly into the
  message text. If you have hotkeys or muscle memory tied to the
  old badge, the new flow is: open slash menu → arrow to skill →
  Enter — the skill command lands in the input ready to type.
- `AgentTool` was renamed to `SubagentTool` (class, directory,
  `TOOL_NAMES.AGENT` → `TOOL_NAMES.SUBAGENT`). Any custom prompts
  or plugin manifests referencing the old symbol must update.
