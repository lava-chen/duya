# v0.1.5

> 2026-06-13 · 62 commits since `v0.1.4`. Headline: the **multi-provider
> migration** (Plans 203 / 204 / 205 / 209) — soft "default provider"
> replaces the single-active-provider model, the settings UI is rebuilt
> as a two-step inline page (picker → edit), and the masked-key bug
> that silently destroyed credentials on edit is fixed. Also lands
> the Agent Mailbox data layer + UI, the new
> `PermissionRequiredError` flow, the BrowserTool zod/v4 schema fix,
> and the `@duya/cli` plugin control plane completion.

## Highlights

- **Multi-provider architecture (Plans 203 / 204 / 205 / 209)** —
  `AppConfig` adds a soft `defaultProviderId` and a `multi-provider-v1`
  migration. The renderer drops the `isActive` filter in 6 places
  and gates visibility on `isDefault + hasApiKey`. The settings
  provider UI is split into a 4-layer architecture (Query →
  Hook-per-concern → Orchestrator → Wiring): `ProvidersSection`
  shrinks from a 1066 LoC monolith to ~80 LoC, with the per-card
  list, add, edit, delete, test, and quota actions moved to a
  `ProviderManagement` orchestrator. Inline edit page replaces the
  `ProviderConnectDialog` modal with a two-step flow: a preset card
  grid (`provider-picker`) → a single-provider form
  (`provider-edit`). The Plan 209 masked-key bug (saving an edit
  without touching the key field persisted the masked placeholder,
  silently destroying the real credential) is fixed with a
  3-state `useApiKeyState` machine and server-side defense in depth
  in `provider-store.upsertLlmProvider`.
- **Agent Mailbox (Plan 202)** — Codex-like in-run instruction
  pipeline. PR1 ships the data layer (`mailbox_*` tables, claim /
  lease / checkpoint, soft interrupt) and the agent server
  hardening. The chat UI gets a `MailboxPanel`, a streaming input
  bar, and the new font-size control.
- **`PermissionRequiredError` flow** — the legacy
  `<tool_use_permission_required>` string sentinel is replaced by a
  typed error class. `BaseTool.execute()` can throw to signal that
  it needs explicit user permission; the executor routes through
  `ToolUseContext.requestPermission`. `PermissionCheckResult` gains
  a `mode` discriminator (`'generic' | 'ask_user_question' |
  'exit_plan_mode'`) so tools with their own dedicated UI declare
  which panel the executor should mount — `AskUserQuestionTool`
  now goes through the dedicated `AskUserQuestionUI` panel rather
  than the generic Allow/Deny prompt.
- **BrowserTool zod/v4 schema fix** — `z.preprocess()` compiles to
  a `ZodPipe` on zod/v4; the previous generator descended via
  `def.schema` and returned the string `'pipe'`, falling through
  to `{ type: 'string' }` and silently telling the LLM that
  `urls` was a string. The generator now walks `def.out` for
  preprocessed fields. Defensive `z.preprocess` wrappers are added
  to `urls`, `useBrowser`, `timeoutMs`, `frameIndex`, `fullPage`,
  `interactiveOnly`, and `maxLength` so the LLM sees the correct
  type even when the tool-call layer stringifies or wraps the
  input. Regression tests pin the fix.
- **Provider list-models endpoint** — `net:provider:models` IPC
  fetches the available model list from a provider's
  OpenAI-compatible `/v1/models` or Ollama `/api/tags`. The
  renderer falls back to the on-disk provider credential when the
  incoming `api_key` is masked or empty, so the chat-side model
  picker works for existing providers without re-typing the key.
- **`@duya/cli` plugin control plane complete (Plan 100)** —
  `duya plugin list` / `info` / `enable` / `disable` / `doctor`
  are shipped. `enable` and `disable` are Phase 7 write ops
  gated on the policy engine; audit events are stamped to the
  unified `control-plane-audit.log.jsonl` with an
  `X-Correlation-Id`.
- **FileEditToolRow (Plan 208)** — `edit` / `write` /
  `create_file` / `str_replace_editor` collapse into a single
  compact card: verb (Edited / Created) + blue clickable filename
  + git-style `+N -M` diff stats. Stats stream live from
  `old_string` / `new_string` during the call and replace with
  the authoritative value from the tool result; number
  transitions animate via framer-motion `AnimatePresence`.
  Click the filename to open in the system default editor.
- **Sub-agent token usage + status** — `runAgent` now captures
  the parent agent's `result` event and writes the `TokenUsage`
  onto the last assistant message before persisting. The
  `SubAgentPanel` honors the parent stream phase; once the
  stream reaches a terminal phase, all sub-agents are reported
  as `completed` rather than stuck on "正在运行…".
- **Chat rendering polish** — `useAdaptiveTypewriter` is
  extracted from `StreamingMessage` into a shared hook so the
  per-block `TextRow` in `ToolActionsGroup` reuses the same
  pacing logic. Text events are now interleaved with tool calls
  in the actions list (each event's accumulated content is
  pushed through), so tool calls render in chronological order
  with the prose. The legacy `StreamingTextContent` and
  cumulative-text `useStreamingText` rendering path are
  removed.
- **Tool result renderers** — `grepRenderer` now consumes the
  new `{ matches: [{ file, line, column, content }], total,
  truncated }` shape: groups by file, shows up to 3 line
  snippets per file, and respects the truncation flag. The
  browser renderer is slimmed to a `canRender` flag; the new
  `BrowserGroup` collapses consecutive browser actions into a
  single toggle that renders the snapshot / compactSnapshot
  content.
- **Gateway permission profile** — IM-channel sessions now
  hardcode `permission_profile = 'default'` instead of falling
  through to the desktop `permissionMode`. The IM channel's
  access control is enforced by the platform-specific pairing /
  allowlist flow, not the desktop permission toggle, so flipping
  the desktop to bypass no longer silently re-promotes IM
  sessions to bypass.

## Agent

### New tools & surfaces
- `PermissionRequiredError` typed flow (above).
- BrowserTool zod/v4 preprocess schema fix (above).
- `net:provider:models` fetcher for the model picker (above).
- `duyaAgent` constructor accepts `runtimeConfig` and feeds it

### Provider
- `LlmProvider` domain foundation (Plan 203 Phase 1):
  `provider-store.defaultId`, `setDefaultLlmProvider`,
  deprecation of the single-active model.
- `provider:setDefaultLlm` / `provider:getDefault` IPC
  handlers; renderer IPC wrappers + DTOs + hooks.
- `multi-provider-v1` migration runs on startup; legacy
  `activeProviderId` users are auto-migrated to
  `defaultProviderId`.
- `sendProviderInit` routes by `thread.providerId` with default
  fallback when a thread has no explicit provider.
- `deleteProvider` clears `defaultProviderId` when the default
  provider is deleted (no dangling reference).
- Runtime convergence: `LlmProvider` ↔ `ApiProvider` adapter,
  capability persistence, query-key React Query integration.
- `provider:activate` is deprecated in favor of
  `provider:set-default`; `duya provider activate` is kept as
  a deprecated alias.

### Reliability & correctness
- `lazySpawnWorkerForCompact` (agent server) builds a
  `providerConfig` that matches the legacy
  `InitMessage.providerConfig` shape — without the helper, the
  worker crashed on `new duyaAgent({...})` with "Model is
  required" on the first chat turn after an agent-server
  restart.
- AskUserQuestionTool permission flow completed
  (`requiresUserConfirmation: true`, `mode: 'ask_user_question'`)
  so the executor mounts the dedicated panel.
- Background sub-agent terminal events surface on the parent
  SSE so `SubAgentPanel` does not stay stuck on "正在运行…".
- `MessageList` / chat rendering: text-only replies, file tool
  rows, base64 paste markers.
- BrowserTool `anyOf` per-operation schema emission so the LLM
  picks the right shape.
- `provider.capability.contextWindow` propagates from the
  capability table to the agent + chat UI.

  preset card grid (`ProviderPickerView`) → single-provider
  form (`ProviderEditView`) with identity / connection /
  advanced sections. Notes field distinguishes multiple

- **FileEditToolRow (Plan 208)** — see Highlights.
- **ToolActionsGroup** — `label: string` migrates to
  `labelKey: TranslationKey` so all action labels are i18n-aware;
  memory tool labels (Read / Saved / Updated / Removed) are
  added. `BrowserGroup` collapses consecutive browser actions
  into a single toggle. The cumulative-text `useStreamingText`

- **Onboarding** — provider form captures the `notes` field
  on first save.
- **Memory** tool labels are wired through to the action row.

## Internals

- `useApiKeyState` is a 3-state machine: `untouched` →
  `replaced` → `cleared`. The 2-state version conflated the
  server-provided mask with user-typed input and was the
  root cause of Plan 209.
- `useApiKeyLink` shares the mask hint with the chat-side
  key entry path.
- `useProviderEditSave` carries an `apiKey?: string`
  contract: `undefined` (keep current) vs `''` (clear) vs
  a new value. `useUpsertProviderMutation` and
  `provider-store.upsertLlmProvider` add the server-side
  mask-rejection defense.
- `src/lib/providers/secret.ts` is a shared
  `isMaskedKey` / `isLikelyRealApiKey` module imported by
  both the renderer and the main process.
- `src/lib/providers/canCheckQuota.ts` is a shared predicate
  for "this provider exposes a quota API" (was inlined in
  `ProviderQuotaView` and `provider-usage`).
- `src/lib/subagent-result.ts` is a shared
  `parseSubAgentToolResult` parser.
- `useAdaptiveTypewriter` is a shared hook (was inlined in
  `StreamingMessage`).
- `ModelInputWithFetch` (settings/forms/shared) is the
  reusable model input used by both the chat-side picker
  and the new `ProviderEditView`.

## Upgrade notes

- `provider:active` / `context.activeProviderId` is deprecated;
  rename to `provider:default` / `context.defaultProviderId`.
  The `multi-provider-v1` migration handles the rename on
  startup — no manual action is required.
- The single-active-provider model (`isActive` filter) is
  gone. Visibility gates on `isDefault + hasApiKey`; the new
  per-row "Set as default" action replaces the legacy
  "Enable" / "In use" toggle.
- Gateway sessions now hardcode
  `permission_profile = 'default'`. If you previously relied
  on `gatewayPermissionMode` / desktop `permissionMode` to
  gate IM-channel sessions, the IM channel is now controlled
  by the platform-specific pairing / allowlist flow instead.
- The `ProviderConnectDialog` modal is gone from the
  settings flow. The settings page now mounts
  `ProviderPickerView` and `ProviderEditView` as inline
  pages; the dialog is still used by the onboarding flow.
- Plan 209's `useApiKeyState` 3-state machine means saving
  an edit without touching the key field now keeps the
  existing credential (was previously the masked-key bug
  that silently overwrote the real key).
