**v0.2.0-beta.2** (2026-07-03) — second beta on the `0.2.x` line.
Since `v0.2.0-beta.1` we are landing the **widget V2 runtime +
self-review surface** (Plan 38, Phase 5), the **`display_content`
persistence boundary** that lets the UI show pasted-document
context while the model sees the original prompt (Plan 102 +
adjacent work), the **right-side panel v4 polish + Recap store**
(Plan 216 + Plan 65), a small **CI / packaging guard bundle** that
adds a build-time agent-bundle verification step and a more
informative `afterPack` FATAL surface, and a **`tool_use_started`
streaming event** that lets the chat UI render a tool card the
instant the model declares the tool name (not after the full input
JSON has arrived).

**Headline**

The headline of this beta is **plan-completion**: the `show_widget`
tool now ships with a **visual self-review pipeline**
(`runVisualSelfReview`) that renders the widget to PNG and asks the
vision model to critique it before yielding the result. The chat
UI gets a dedicated `WidgetRenderer` mounted in
`MessageList`/`MessageItem` and a new `WidgetSanitizer` +
`widget-css-bridge` pair in `@duya/conductor/elements` so the
widget code stays inside a CSS-isolated DOM tree. The widget
guidelines prompt is updated to teach the model the new self-
review contract.

**Highlights**

- **`show_widget` visual self-review (Plan 38, Phase 5)** — new
  `packages/agent/src/tool/WidgetRenderer/` ships `HeadlessWidgetRenderer`
  (a persistent chromium render-to-PNG service) and a high-level
  `runVisualSelfReview` that runs "render → vision critique →
  rewrite" inside the agent loop. The result is delivered to the
  LLM as a synthetic second `tool_result` with the same
  `tool_call_id` so downstream consumers (LLM, SSE channel, DB)
  see both the widget code and the critique under one tool call.
- **`ToolResult.extraResult` deferred second result** —
  `StreamingToolExecutor` keeps a reference to the deferred promise
  on a `ToolResult` and, after the main result has been delivered,
  awaits it and yields a synthetic second `tool_result`. This is
  the public mechanism the widget self-review uses.
- **`HeadlessWidgetRenderer` lifecycle** —
  `disposeHeadlessWidgetRenderer()` releases the persistent
  chromium instance when the agent process shuts down.
- **`@duya/conductor/elements` widget primitives** — new package
  exports `widget-sanitizer` (CSS-only sanitization for widget
  markup, prevents script injection from LLM-generated widget
  code) and `widget-css-bridge` (theme-aware CSS variable bridging
  so widgets stay readable in both light and dark mode).
- **`stripPastedContentMarkers`** — new utility in
  `packages/agent/src/utils/pasted-content.ts` that strips the UI
  marker block before the model sees the prompt. Used by `duyaAgent`
  to keep the model-facing `content` clean while the UI persists the
  marker block separately.
- **`displayContent` persistence (Plan 102)** — `messages` table
  gains a `display_content` column (migration 35). UI markers,
  pasted-document context, and synthetic compaction context are
  now persisted separately from the model-facing `content` so the
  original prompt is what the model sees on every replay. The
  `replaceMessages` comparison uses `prompt` (not `displayContent`)
  to prevent the model-facing path from accidentally being
  rewritten by the UI-only field.
- **`tool_use_started` streaming event** — `AnthropicClient` and
  `OpenAIClient` now yield a `tool_use_started` SSE event as soon
  as the model declares the tool `name` (not after the full input
  JSON arrives). The chat UI uses this to render the tool card
  immediately; `input` is patched in as the JSON deltas arrive.
- **Provider safety-filter error class** — `APIErrorType` gains
  `PROVIDER_SAFETY_FILTER` (`'provider_safety_filter'`), classified
  from `new_sensitive` / `sensitive content` / `content policy`
  error strings. `formatErrorForDisplay` explains that previous
  tool work is kept and the user can rephrase or switch models.
- **`AgentMailbox` display_content wiring** — the chat-start
  payload accepts `displayContent`; `messageRowToMessage` reads the
  new column and propagates it to the renderer.
- **Right-side panel v4 polish + Recap store (Plan 216 + 65)** —
  `usePanel` exposes the multi-tab registry; `BrowserPanel`,
  `PanelFileTreeSplit`, and `registry` are wired to the new
  `tabs / openPage / openOrActivatePage / activateTab / closeTab`
  API. New `recap-store.ts` lays the data-layer foundation for
  Plan 65 (Session Recap). `chat-file-links.ts` gains
  `isOfficeFile` + `SIDEBAR_PREVIEW_EXTENSIONS` so the panel can
  route `.docx` / `.pptx` / `.xlsx` to `OfficePanel` and the rest of
  the preview surfaces to `FilePreviewPanel`.
- **`AgentServer` agent-message-types** — `agent-message-types.ts`
  gains `displayContent?: string` so the streaming payload matches
  the persisted shape.
- **CI / packaging guards** —
  - `ci(release)`: new step between `electron:build` and
    `electron:pack` that verifies
    `release/win-unpacked/resources/agent-bundle/agent-process-entry.js`
    and `BashTool/BashWorker.js` exist. Missing bundle fails the
    workflow before `electron-builder` produces an installer that
    crashes on first launch.
  - `afterPack` FATAL surface: when an Electron build resolves the
    agent bundle from the source tree instead of the packaged
    `resources/agent-bundle/`, the FATAL log now reports both paths
    and the resolved value, so an operator can immediately tell
    whether the running binary is reading its bundle from
    `resources/` (correct) or from a stale `npm run electron:dev`
    artifact.
- **`electron:dev` script fix** — `package.json:electron:dev` now
  runs `npm run bundle:agent` before `node scripts/build-electron.mjs`
  so the dev launch path matches the production bundle path and
  `npm run electron:dev` no longer runs against a stale `dist/`.

**Upgrade notes**

- `messages.display_content` (migration 35) is added on the next
  start; existing rows get `NULL` and the column is read-only on
  the UI side (write path is the agent process, not the renderer).
- `ToolResult.extraResult` is additive — older consumers that only
  inspect `content` / `error` are unaffected.
- `tool_use_started` is an additive SSE event; older consumers
  that only handle `tool_use` / `tool_result` will simply ignore
  the early-start event (the eventual `tool_use` with the full
  `input` still arrives).
- `PROVIDER_SAFETY_FILTER` is an additive error class; older
  consumers that fall through to `UNKNOWN` for unknown error types
  will still see the original error string.
- `package.json:electron:dev` now requires `bundle:agent`. If you
  keep a custom dev script, run `npm run bundle:agent` first.
- `isOfficeFile` and `SIDEBAR_PREVIEW_EXTENSIONS` are additive
  exports from `chat-file-links.ts`.
