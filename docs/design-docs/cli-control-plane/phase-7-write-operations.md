# Phase 7: Write Operations Permission Model

> **Status**: Design Document — Locks the rules for CLI write operations
> **Created**: 2026-06-03
> **Scope**: Defines the policy for `duya skill enable/disable`, `duya plugin
> enable/disable`, `duya session delete`, and other future write commands.

---

## 1. Default Posture (Locked)

**All write operations are denied by default** in non-interactive / external agent scenarios. The CLI does NOT expose write operations that:

- Modify API keys (e.g. `provider set-key`)
- Install or remove plugins (`plugin install / remove`)
- Add or remove MCP servers (`mcp add / remove`)
- Delete sessions (`session delete`)

The following **low-risk reversible** operations MAY be exposed in v0 IF the permission model below is implemented:

- `duya skill enable <id>` / `duya skill disable <id>`
- `duya plugin enable <id>` / `duya plugin disable <id>`

These are the only write operations the CLI will support in v0.

---

## 2. Permission Model (Locked)

### 2.1 Interactive Mode (default)

- CLI is **interactive** when stdin is a TTY (not piped, not redirected)
- Interactive write operations prompt for confirmation
- Default answer: `N` (no)
- `duya skill disable <id> --yes` skips the confirmation prompt
- Failure mode: if the user types anything other than `y` or `yes`, the operation is aborted with exit code 1

### 2.2 Non-Interactive Mode (scripted, agent-driven)

- CLI is **non-interactive** when stdin is NOT a TTY
- All write operations REQUIRE `--yes` flag
- If `--yes` is not provided:
  - Error: `interactive_required` with hint "pass --yes to confirm"
  - Exit code 3 (specific to non-interactive denial)
- If `--yes` IS provided:
  - Operation proceeds; audit log entry written
  - Exit code 0 on success, 1 on failure

### 2.3 Rate Limiting (v0)

- v0: no rate limit; users / scripts are responsible for throttling
- Future (v1+): add per-minute budget enforced at the CLI server side
- Audit logs are the v0 visibility mechanism

---

## 3. Audit Log (Locked)

### 3.1 Where Audit Logs Are Written

- **NOT in the CLI process.** Audit logs are written by the main process to a unified control-plane audit store.
- Storage location: `<userData>/control-plane-audit.log.jsonl`
- Format: newline-delimited JSON, one record per line

### 3.2 What Is Logged

| Event | Required fields |
|-------|-----------------|
| `skill.enable` | `id`, `ts`, `invokedBy: cli`, `correlationId?` |
| `skill.disable` | `id`, `ts`, `invokedBy: cli`, `correlationId?` |
| `plugin.enable` | `id`, `ts`, `invokedBy: cli`, `correlationId?` |
| `plugin.disable` | `id`, `ts`, `invokedBy: cli`, `correlationId?` |

### 3.3 What Is NOT Logged

- API keys, bearer tokens, secrets of any kind
- Session message content, system prompts
- Full request/response bodies for the operation
- User filesystem paths (only op ids)

### 3.4 Correlation

- The CLI generates a `correlationId` (random UUID v4) per write invocation
- The CLI passes it via `X-Correlation-Id` header
- The main process records it in the audit log entry
- This allows operators to trace the action back to the CLI invocation

## 4. CLI Command Surface (v0)

### 4.1 Skill Write Commands

```
duya skill enable <id>    [--yes]
duya skill disable <id>   [--yes]
```

- `id` is the public skill id (`bundled:<n>` / `plugin:<id>:<n>` / `user:<n>`)
- The CLI calls `POST /v1/skills/:id/enable` (or `/disable`)
- HTTP endpoint records the audit log entry
- Returns 200 with the new effective state

### 4.2 Plugin Write Commands (v0, optional)

```
duya plugin enable <id>   [--yes]
duya plugin disable <id>  [--yes]
```

- Same pattern as skill

### 4.3 Forbidden Write Commands (v0)

| Command | Reason |
|---------|--------|
| `duya provider set-key` | API key entry; high-risk; GUI only |
| `duya plugin install <path>` | Installs new plugin; high-risk; GUI only |
| `duya plugin remove <id>` | Removes installed plugin; high-risk; GUI only |
| `duya mcp add <config>` | Adds MCP server; high-risk; GUI only |
| `duya mcp remove <id>` | Removes MCP server; high-risk; GUI only |
| `duya session delete <id>` | Deletes session; data-loss risk; GUI only |

These commands do not exist in v0. Future versions MAY add them after a separate audit and permission model expansion.

---

## 5. Server Endpoints (v0)

```
POST /v1/skills/:id/enable
POST /v1/skills/:id/disable
```

- Both require `X-Correlation-Id` header (optional; auto-generated by CLI)
- Both record audit log entry in the main process
- Both return `{ skill: <updated-DTO> }` with the new effective state

---

## 6. Tests Required (v0)

1. `duya skill disable <id>` in interactive mode: prompts for confirmation; abort on `n`
2. `duya skill disable <id> --yes` in non-interactive mode: proceeds
3. `duya skill disable <id>` without `--yes` in non-interactive mode: exit 3 with `interactive_required`
4. Audit log entry written with correct `id`, `ts`, `invokedBy: cli`, `correlationId`
5. Audit log does NOT contain API keys, secrets, or session content
6. Token / auth / runtime error paths: clear error and non-zero exit
7. Shadowed id (winner has higher precedence) returns `skill_not_found`
8. `duya doctor` reports the new audit log file exists and is valid JSONL
